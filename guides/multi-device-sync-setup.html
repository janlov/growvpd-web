Multi-Device Sync: Two Phones, Same Diary

If you ever wished your partner could log a feeding from their phone while you’re at work, or wanted your bedside tablet to show the same VPD chart as your daily phone, multi-device sync is for you. Pair two or more phones (or tablets) to a single shared diary — encrypted, LAN-first, no cloud account required.

1. Why you might want this

Common scenarios:

  • Couple sharing a grow. Both partners can log feedings, snap photos, see the same calendar. No one is left in the dark.
  • Work phone + home phone. Read on commute, edit at home. Hand-off seamless.
  • Daily phone + 24/7 hub phone. Hub keeps automation running while you’re out; both see live data.
  • Multi-tent at different physical locations. Tablet at site A logs entries; primary phone at site B has the same diary.
  • Backup confidence. If one phone dies, the other has everything.

2. How it works under the hood

The architecture in plain English:

  1. Each device generates an Ed25519 keypair. Stored in the Android Keystore, never leaves the device.
  2. Pairing exchanges public keys via QR. After scan, both devices know each other’s identity.
  3. Diary edits are HLC-timestamped. A Hybrid Logical Clock combines wall time + a per-device counter so we can order edits even if devices were offline.
  4. Sync messages use LWW (last-writer-wins) merge. The HLC tells us which edit is newer; the newer one wins.
  5. Transport is encrypted with libsodium box. XChaCha20-Poly1305. We literally cannot decrypt it on our cloud relay.
  6. Discovery is via mDNS (Bonjour-style) on LAN. Each paired device advertises _growvpd._tcp.local. and finds peers automatically.
# Wire-level sync message (simplified, post-decryption)
{
  "type": "diary_entry_upsert",
  "hlc": "1714200000000:42:a3f5",
  "device_id": "phone_a",
  "payload": { "id": 1234, "growId": 5, "vpd": 1.1, ... }
}

3. Privacy: end-to-end encrypted, no central server

Every byte that traverses the sync channel — whether over LAN or our optional cloud relay — is encrypted with libsodium box. The encryption keys are derived from each device’s Ed25519 keypair via X3DH-lite handshake during pairing.

What this means concretely:

  • Our cloud relay (if you opt in) sees encrypted blobs only. We can’t read them.
  • If you don’t opt in to cloud relay, sync stays purely LAN. Zero bytes touch our infrastructure.
  • Even if Android leaks your APK or someone takes the phone, the sync keys are in the Android Keystore (StrongBox/TEE on supported devices) — not extractable without root + significant effort.

For the truly paranoid: the source code for the entire sync layer lives in data/sync/ in our codebase. We’re happy to share private repo access with security researchers (DM on Discord).

4. Pairing flow: QR scan

Step 1

Open Multi-Device Sync on Phone A

Settings → Multi-Device Sync → Pair new device. Phone A:

  1. Generates an Ed25519 keypair if it doesn’t have one (one-time, instant).
  2. Stores it in the Android Keystore.
  3. Computes a short-lived (60-second) pairing token.
  4. Renders a QR code containing: device id, public key, current LAN IP, pairing token.
Step 2

Scan from Phone B

On Phone B: Settings → Multi-Device Sync → Scan QR. Camera opens, point at Phone A’s screen. Within ~1 second the QR is decoded.

Phone B then:

  1. Generates its own Ed25519 keypair (one-time).
  2. Performs X3DH-lite handshake using Phone A’s public key + the pairing token.
  3. Connects to Phone A’s IP on the sync TCP port.
  4. Both devices show a confirmation popup: “Pair with [Phone A name]?”.
Step 3

Confirm on both devices

Tap Confirm on both. The pairing token is consumed (cannot be reused). Both devices store each other’s public keys in their PairedDevice Room table.

Step 4

Initial backfill

The newer (less-data) device pulls a complete snapshot of the older device’s data. Backfill duration:

  • Empty new device: ~30 seconds for a year of diary.
  • Both have data, merge needed: ~2–5 minutes.
  • One has 1000+ photos: ~10–15 minutes (photos transfer separately).

You can use either phone during backfill; sync runs in the background.

Step 5

Done. Live sync from now on

Every diary edit, photo upload, automation rule change propagates automatically. You can manually trigger via Settings → Multi-Device Sync → Sync now (handy after long offline periods).

5. When sync triggers

Sync runs in three modes:

  1. Live (event-driven). When you save an entry, take a photo, change a rule — immediately broadcast to paired devices that are reachable. Latency on LAN is ~200–500 ms.
  2. App-open catch-up. When you open the app after being closed, do a quick handshake with paired devices to discover what changed.
  3. Periodic background. Every 15 minutes (less aggressive on battery saver), check for missed updates. Mainly for cases where live broadcast missed a peer (e.g. peer was off Wi-Fi).

6. Conflict resolution: last-writer-wins via HLC

What happens if both phones edit the same diary entry while offline, then reconnect?

The Hybrid Logical Clock tags every edit with a triplet: (wall_time_ms, counter, device_id_hash). When two edits collide:

  1. Compare wall_time_ms first. Newer wins.
  2. If equal (very rare), compare counter.
  3. If still equal, device_id_hash breaks the tie deterministically.

This means “last writer wins” is well-defined even with clock drift. Both devices end up with the same final state without coordinator.

What you see in the UI

Most of the time, conflicts resolve invisibly. If we detect a meaningful semantic conflict (e.g. Phone A logged “PPM 800” and Phone B logged “PPM 1200” for the same date in the same minute), we flag the entry with a small conflict icon and let you tap to see both versions and choose.

For non-semantic conflicts (different fields edited on the same entry), we merge field-by-field by HLC.

7. LAN sync via mDNS

LAN sync is the default and the privacy gold standard. Both devices need to be on the same Wi-Fi (same subnet, technically). The discovery flow:

  1. Each paired device advertises an mDNS service: _growvpd._tcp.local. on port 8443.
  2. Other paired devices browse that service type and discover peers by name.
  3. When a peer is discovered, they open an encrypted TLS-over-Noise channel.

Latency is excellent (~200 ms for typical edits). Bandwidth is local Wi-Fi speed; you’ll never feel a transfer except for huge photo dumps.

8. Cloud relay fallback (Pro)

If both phones aren’t on the same Wi-Fi at the same time, sync stalls. The fix is our optional cloud relay.

Enable in Settings → Multi-Device Sync → Use cloud relay when off LAN. From then on:

  • Each device opens an outbound MQTT connection to relay.growvpd.pro.
  • When you make an edit, the encrypted message goes to the relay.
  • The relay holds it (briefly) until the destination peer connects.
  • Destination peer pulls the encrypted blob, decrypts locally.

The relay is opt-in and minimalist by design:

  • Our infrastructure: stateless. Messages are deleted immediately after delivery.
  • Encryption: same libsodium box as LAN. Relay sees ciphertext only.
  • Throughput: minimal. We don’t store images on the relay; only metadata + small text payloads.

9. What gets synced (and what doesn’t)

Synced

  • Diary entries (everything: temp, RH, VPD, notes, photo refs)
  • Strain catalogue and lineage
  • Equipment profiles and tent setups
  • Smart device list (which devices belong to which tents)
  • Automation rules
  • Sensor readings (last 30 days; older is summarised)
  • Harvest data and curing logs
  • Photos (transferred separately, lazy)

Not synced (deliberate)

  • Smart device credentials. Tuya client_secret, Mars Hydro password, etc. Each device pairs platforms independently for security — if one phone is compromised, credentials don’t leak to others.
  • Local app settings. Theme, language, biometric lock, notification preferences. These are device-personal.
  • App lock PIN. Set independently per device.
  • Camera permissions etc. OS-level and per device.

10. Troubleshooting

“QR scan times out”

The pairing token in the QR is valid for 60 seconds. If you took longer (e.g. couldn’t find Settings on Phone B), regenerate the QR on Phone A and scan promptly.

“LAN discovery fails (mDNS blocked)”

Some routers block mDNS by default. Most common offenders:

  • TP-Link Deco mesh: enable “mDNS forwarding” in advanced settings.
  • Asus AiMesh: Settings → LAN → IPTV → enable IGMP snooping or set “mDNS=enabled”.
  • Eero: usually fine; if not, factory reset and re-setup.

As a workaround, the cloud relay (section 8) sidesteps mDNS entirely.

“Diary entries appear twice”

Rare. Caused by both devices editing while offline + a HLC tie that wasn’t resolved cleanly. Tap the conflicting entry → Resolve conflict → pick which version wins. The unchosen version is archived (recoverable from Settings → Diary → Conflict history).

“Sync stalled, paired devices show ‘offline’”

Three causes:

  • Both devices are off LAN (not on same Wi-Fi). Either reconnect, or enable cloud relay.
  • Battery saver killed the foreground service on one device. Apply Hub-Mode-style tweaks (see that guide).
  • App not opened recently — periodic background sync hasn’t fired. Tap “Sync now”.

“Photos take forever to sync”

Photos are transferred lazily over LAN at ~1 MB/s (depending on Wi-Fi). A 5 MB photo takes ~5 seconds; a 100-photo backfill takes ~10 minutes. Cloud relay does not transfer photos; only their metadata. Photos always sync via direct LAN or peer-to-peer over WebRTC fallback.

“I unpaired by mistake; can I recover?”

Yes. Re-pair via fresh QR. Backfill brings everything from the other device. Worst-case data loss is anything that was edited on the unpaired device after unpairing — bring it back into the fold within 30 days and HLC will reconcile.

11. FAQ

Q: How many devices can I pair?

Practically unlimited. We’ve tested 5 devices in a single sync mesh without slowdown. Overhead per device is small (one X3DH handshake at pair time).

Q: Can I pair iOS and Android?

Yes (when iOS multi-device ships, currently in beta with limited push reliability). Same QR pairing flow. Sync works the same way; iOS just can’t do 24/7 background reliably (see Hub Mode guide).

Q: Does sync work without internet at all?

Yes, on LAN. Both devices on same Wi-Fi, no internet, full sync works. The cloud relay only matters when devices are on different networks.

Q: What if I want to remove a paired device permanently?

Settings → Multi-Device Sync → Paired devices → tap device → Unpair. The other device’s public key is revoked; future inbound messages from it are rejected. The unpaired device still has its local data but won’t receive your future updates.

Q: Are diary photos visible on every paired device?

Yes once they sync (lazy transfer). Latency: ~10 s per photo on LAN, longer over cellular. The newest photos are prioritised so your timeline stays useful.

Q: Can I have a write-only device that doesn’t pull data?

Not currently — sync is bidirectional by design. If you want a logging-only device, just don’t pair it; export data manually instead.
